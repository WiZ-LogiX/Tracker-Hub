## 1. Project Overview

**PeleCanon** is a multi-tenant SaaS for **furniture pricing, quotation, and production tracking**, focused on the Egyptian market (Arabic-first + RTL, EGP currency, 14% VAT).

- **Repository root**: `package.json` → `pelecanon`, `"type": "module"`.
- **Domain**: B2B/B2C furniture workshops (kitchens, wardrobes, bedrooms, tables, office furniture).
- **Lifecycle of an order**: Catalog → Quote (draft/sent/accepted/rejected/converted) → Invoice (paid/unpaid) → Production Order (9-stage workflow with photo logs, worker assignments, QC) → Customer-facing tracking page → Customer notifications (WhatsApp via n8n webhook).
- **Identity anchor**: Unified `PLC-XXXXX` ID (5 random alphanumeric chars) ties Quote → Invoice → Order chain. Generated client-side in `src/lib/numbering.ts` (`generatePLCId()`) and via the `generatePLCNumber` server fn in `src/lib/plc.functions.ts`.
- **Current architectural posture (per `.lovable/STATUS.md`)**: Phase 0 bootstrap 100% ✓, Phase 1 tenancy DB 100% ✓, Phase 2 custom auth 100% ✓, Phase 3 R2 ~70%, Phase 4 quoting 100% ✓, Phase 5 production tracking 100% ✓, Phase 6 notifications 80%, Phase 7 Neon migration **partially cut over** (client still uses Supabase Postgres; Drizzle adapter + Neon HTTP driver wired but **not** active).

## 2. Business Purpose

PeleCanon solves three problems for furniture workshops:

1. **Transparent pricing** — Provide a central, deterministic pricing engine (`src/lib/pricing.ts`, `src/lib/pricing/engine.ts`) that surfaces every cost component (base + material + finish + veneer + accessories + labor/wastage/overhead + margin + luxury/complexity/rush).
2. **Production visibility** — Give customers a public tracking page (`/track?ref=ORD-XXX`) and provide admins a kanban-by-stage view (`src/routes/admin/orders.tsx`) with worker assignments and QC.
3. **Multitenancy** — Each "company" is a tenant with its own branding (`src/integrations/supabase/icp-charts.tsx` references tenant.tenant_id as the brand isolation key). Phase 1 already wired 33 business tables to `tenant_id` with RLS (`supabase/migrations/20260612_tenancy_v1.sql`).

The market edge: 9-stage production visibility (deposit received → design approved → cutting → assembly → finishing → QC → ready → delivered → completed), and a config-driven pricing DSL that lets admins tune formulas without code changes (`src/routes/admin/pricing-rules.tsx`).

### 3.2 Request lifecycle

1. Edge request → `src/server.ts` (Cloudflare Workers entry).
   - `getServerEntry()` dynamically imports `@tanstack/react-start/server-entry` (cached).
   - `normalizeCatastrophicSsrResponse()` catches the **h3 swallowed SSR error** pattern: when a 3rd-party lib throws inside h3, the Response body becomes `{"unhandled":true,"message":"HTTPError","status":500}`. The error-capture (`src/lib/error-capture.ts`) hooks `globalThis` `error` and `unhandledrejection` listeners to preserve the real `Error`. The wrapper substitutes a branded `renderErrorPage()` HTML page when this shape is detected. (Comment: "h3 swallows in-handler throws into a normal 500 Response" in `src/server.ts:40`).
2. TanStack Start middleware (`src/start.ts`):
   - `errorMiddleware` — catches remaining errors, returns branded HTML.
   - `csrfMiddleware` — `createCsrfMiddleware({ filter: (ctx) => ctx.handlerType === "serverFn" })` — only server-fn calls are CSRF-checked.
   - `attachSupabaseAuth` — `functionMiddleware` (client-side attacher) which sets `Authorization: Bearer <token>` on every server-fn RPC call. Implemented in `src/integrations/supabase/auth-attacher.ts` (gen'd boilerplate).
3. Server functions use `.middleware([requireSupabaseAuth])` from `src/integrations/supabase/auth-middleware.ts`.
   - The middleware extracts `Authorization: Bearer <token>` from headers, verifies via `supabase.auth.getClaims(token)`, populates `context = { supabase, userId, claims }`, and throws a string error if missing.
4. Database access flows:
   - **Browser**: `supabase.from('table')` (RLS-enforced; client at `src/integrations/supabase/client.ts`).
   - **Server (admin)**: `supabaseAdmin.from('table')` (service-role, RLS bypass; client at `src/integrations/supabase/client.server.ts`).
   - **Server (user context)**: middleware-injected `supabase` (RLS-enforced).

### 3.3 Multi-tenant data path

- RLS policy functor: `is_tenant_member(_tenant_id uuid, _roles tenant_role[])` (SECURITY DEFINER, `search_path=public`, `supabase/migrations/20260612_tenancy_v1.sql`).
- Every business table has `tenant_id uuid NOT NULL` + immutability trigger.
- Application glue: `src/lib/tenant-context.ts` exposes `TENANT_ROLES`, `canWrite(role)`, `canDelete(role)`, `requireRole(ctx, allowed)`.
- `src/db/tenancy-schema.ts` — Drizzle table definitions for `tenants` + `tenant_members` (schema mirror).
- Note from `.lovable/STATUS.md`: `user_roles` was dropped; legacy FKs to it are broken; `useAuth.tsx` is now in a transition state (uses `tenant_members` directly + bootstraps a fresh role row).

## 5. Core Components

### 5.1 Pricing engine ([`src/lib/pricing.ts`](src/lib/pricing.ts), [`src/lib/pricing/engine.ts`](src/lib/pricing/engine.ts))

- Two engines coexist:
  1. **`calculateLine` + `calculateQuoteTotals`** — classic deterministic engine. Inputs: base price, dimension (m² or linear m), qty, material rate per unit, finish pct/fixed modifiers, accessories total, labor/wastage/overhead/margin %. Outputs labels including `baseCost`, `materialCost`, `finishCost`, `accessoriesCost`, `subtotalBeforeOverhead`, `laborAmount`, `wastageAmount`, `overheadAmount`, `marginAmount`, `unitPrice`, `lineTotal`. Discount handling: percentage or fixed + `max_value` cap. VAT default 14%.
  2. **`runFormula(formula, selections, factors, ruleVersion)`** — JSON-DSL interpreter. Steps: `add (which bucket)`, `snapshot (label)`, `mul_pct (factor key, against snapshot or running, ?add)`. Default formula archives labor/wastage/overhead/margin + luxury/complexity/rush as optional per-item overrides (`src/lib/pricing/engine.ts`).
- Quote totals machinery computes `subtotal`, `discountAmount`, `vatBase`, `vatAmount`, `total`.
- `formatEGP(n)` always uses `Intl.NumberFormat('ar-EG', { style: 'currency', currency: 'EGP', maximumFractionDigits: 2 })`.

### 5.2 Order workflow machine ([`src/routes/admin/orders.tsx`](src/routes/admin/orders.tsx), [`src/lib/stages.ts`](src/lib/stages.ts))

```
deposit_received → design_approved → cutting → assembly → finishing → quality_check → ready_for_pickup → delivered → completed
```

- `nextStage(s)` returns null at the last stage.
- QC gate: `canAdvance(order)` requires a passing `qc_inspections` row for `quality_check` or the upgrade is blocked (UI shows toast error `orders.advanceGate`).
- Worker assignments tracked per stage: `pending → in_progress → completed`. Each assignment has `started_at` + `finished_at`.
- Advance side-effects: insert `production_logs` row, optionally send `notify({ data: { event, entityType:'order', entityId, extra } })` via `sendNotification`, and (if `next === 'delivered'`) stamp `delivered_at` on the order.

### 5.3 Generic CRUD widget ([`src/components/admin/GenericCrud.tsx`](src/components/admin/GenericCrud.tsx))

- Bypass tables have service fn paths registered so reads + writes go through `supabaseAdmin` (RLS bypass). `TABLES_WITH_BYPASS` = product_templates, materials, suppliers, finishes, veneers, accessories, pricing_factors, wastage_rules, pricing_rules, workers, discounts.
- Hooks are **unconditionally** allocated (`useServerFn` for every bypass table every render) — comment warns that conditional hook allocation flips hook order across renders and crashes Dialog mounts.
- `PROTECTED_INSERT_FIELDS` (id, tenant_id, created_at, updated_at) are filtered out before saving.
- `auth/loading` gates render so the table never flashes before tenant is resolved.

### 5.4 Photo upload pipeline ([`src/components/photo-uploader.tsx`](src/components/photo-uploader.tsx), [`src/components/photo-grid.tsx`](src/components/photo-grid.tsx), [`src/lib/useSignedR2Urls.ts`](src/lib/useSignedR2Urls.ts))

- Flow:
  1. Photos → `getR2BatchUploadUrls` (server fn) → returns `{ uploads: [{key, uploadUrl, publicUrl?}] }`.
  2. **Real OPTIONS preflight** (`probePreflight`) — fires an `Access-Control-Request-Method: PUT` preflight before the actual PUT. The comment notes: "HEAD on a signed-PUT URL is a CORS *simple* request — browsers don't preflight it and don't send Content-Type/Authorization, so it succeeds even when the PUT that follows would be blocked."
  3. Direct `fetch(uploadUrl, { method: 'PUT', body, headers: { 'Content-Type': <file.type> } })` to R2.
  4. Insert into `production_photos` with the public URL.
- `useSignedR2Urls(urls[])` keys cache by sorted-joined URL list, `staleTime = 25min`, `gcTime = 30min` — deliberately shorter than the 25-minute signed URL TTL (25min matches signed TTL, gc 30 is the safety margin).
- `PhotoGrid` groups by `ORDER_STAGES` order.

### 5.5 Auth pipeline ([`src/lib/useAuth.tsx`](src/lib/useAuth.tsx), [`src/lib/auth.functions.ts`](src/lib/auth.functions.ts), [`src/routes/auth.tsx`](src/routes/auth.tsx))

- **Username login**: `proxyEmailFor(username) = ${username}@pelecanon.local` (RFC-compatible synthetic email so Supabase Auth still has a unique `email` column). Username regex: `/^[a-z0-9._-]+$/i`, min 3, max 32.
- First login: `ensureBootstrapAdmin` server fn creates `pelecanon` tenant + `admin/admin` user with `owner` tenant role if not already present.
- After login, `setSession({access_token, refresh_token})` persists token on the client side and triggers `onAuthStateChange` → bootstrap call.
- Bootstrap (`bootstrappedForRef` token): runs once per user. If the bootstrap tenant call fails, UI shows `bootstrapFailed` panel with a Retry button.
- 6 admin flows under `/admin/team`: createAppUser, listAppUsers, setUserStatus(active|disabled), resetUserPassword, updateUserAvatar, getCurrentUser (informational).

### 5.6 Tenant context ([`src/lib/tenant-context.ts`](src/lib/tenant-context.ts))

- `TenantRole = 'owner' | 'admin' | 'sales' | 'worker' | 'viewer'`.
- Helpers: `canWrite(role)` (owner/admin/sales → write), `canDelete(role)` (owner/admin only).
- `TenantContext = { userId, tenantId, role }` for downstream server fns to consume.

### 5.7 Multi-tenant server functions

- Catalog fns (`src/lib/catalog.functions.ts`) bypass RLS through `supabaseAdmin`.
- Server fns that must respect RLS go through `supabase` injected by `requireSupabaseAuth` middleware (`src/integrations/supabase/auth-middleware.ts`).
- Helper: `resolveTenantId(userId)` in `src/lib/attachments.functions.ts:35-45` reads `tenant_members.tenant_id` ordered by created_at asc (canonical "first tenant" semantics).

## 6. Data Flow

### 6.1 Quote → Order flow (most important business path)

```
[Admin] /admin/quotes/new or /configurator
[Browser] compute total via runFormula / calculateLine using local pricing_factors + materials
[Browser] inserts into `quotes` (quote_number = PLC-XXXXX, customer_id, total, snapshot)
[Browser] inserts into `quote_items` (per-item breakdown JSON, accessories)

[Admin] /admin/quotes/$id → click "تحويل لفاتورة وأمر إنتاج"
[Browser] invoke createInvoiceFromQuote({ data: { quoteId, customerId, plcId } })
   └─ server: resolves tenant_id, copies totals + snapshot, sets status='converted'
[Browser] invoke createOrder({ data: { quoteId, invoiceId, customerId, plcId } })
   └─ server: links via quote_id/invoice_id, sets current_stage='deposit_received', expected_delivery=now+30d
[Browser] optional: notify({ data: { event: 'order_opened', entityType: 'order', entityId } })

[Admin] /admin/orders → advance stages ... loop ...
[Customer] /track?ref={quote.quote_number || order.order_number || invoice.invoice_number}
   └─ server: getPublicTrackingByRef looks up by order_number first, then quote, invoice, quote_request
```

### 6.2 Photo upload to R2

```
UI PhotoUploader.handlePick(files)
  → getR2BatchUploadUrls({data:{files,entityType:'production-photos',entityId:orderId}})
    ← server signs each: returns {uploads:[{key,uploadUrl,publicUrl}]}
  → probePreflight(uploadUrl)        // OPTIONS preflight, raises on 4xx
  → fetch(uploadUrl, {method:'PUT', body, headers:{Content-Type}})
[Browser] supabase.from('production_photos').insert(rows)
```

### 6.3 Avatar upload

```
AvatarUploader.onPick(file)
  → resizeImage(file) → webp ≤512×512 (OffscreenCanvas or fallback canvas)
  → getR2BatchUploadUrls({files:[{filename,contentType:'image/webp'}],entityType:'avatars',entityId:userId})
  → fetch(uploadUrl, PUT, blob)        // direct
  → updateUserAvatar({data:{userId,avatarKey}})
```

### 6.4 Notification dispatch

```
UI click on a notification trigger
  → sendNotification({ data: { event, entityType, entityId, extra?, language? } })
      ├─ load entity (quote/order/invoice) via supabaseAdmin
      ├─ read template from notification_templates (locale → fallback 'en')
      ├─ render subject + body with variable substitution
      ├─ POST to N8N_NOTIFY_WEBHOOK_URL with optional X-Lovable-Token
      └─ insert notification_log row (status: sent/failed/skipped)
```

## 9. Authentication & Authorization

- **Provider**: Supabase Auth (email/password). Google OAuth is **explicitly disabled** per `.lovable/neon-migration-plan.md`.
- **Credentials**: Username + password only. Admin-managed; no self-signup.
- **Synthetic email**: `proxyEmailFor(username) = ${username.toLowerCase()}@pelecanon.local` (RFC-compatible) — preserves a unique `auth.users.email` per row while hiding the email shape from the UI.
- **Password rules**: min 6, max 128 chars (enforced in `Username = z.string().trim().min(3).max(32).regex(/^[a-z0-9._-]+$/i, ...)`; `Password = z.string().min(6).max(128)`).
- **Session**: Persisted via `supabase.auth.setSession({access_token, refresh_token})` in `src/routes/auth.tsx:60`. Cookies via the configured `pelecanon-auth-token` storage key.
- **Tenant bootstrap**:
  - First login triggers `ensureBootstrapAdmin` (creates `pelecanon` tenant + admin/admin if missing).
  - `bootstrapMyTenant({...})` self-heals: fetches tenant + inserts one row in `tenant_members` per user, picks first membership as "primary" for the next request.
- **Middleware**:
  - `attachSupabaseAuth` (client) → forwards bearer token (`src/integrations/supabase/auth-attacher.ts`).
  - `requireSupabaseAuth` (server) → validates the token via `supabase.auth.getClaims(token)` (`src/integrations/supabase/auth-middleware.ts`).
- **Authorization model**:
  - **App role**: via `tenant_members.role` (Phase 1+) — `isStaff = memberships.some(m => m.role !== 'viewer')` gate in `src/lib/useAuth.tsx`.
  - **Server fn authorizations**:
    - `applyWastageRulesMigration`: requires owner/admin role in any tenant.
    - `createAppUser`: requires owner/admin in caller's first tenant; resolves `tenantId` from membership.
    - `getR2UploadUrl`/batch: requires tenant — keys are `tenantId/entityType/entityId/<hash>.<ext>` so isolation follows even if RLS slips.
    - `deleteR2Object`: requires `key` to start with `${session.tenantId}/`.
    - `getR2DownloadUrl`: same path-prefix check.
  - **RLS (DB layer)**: every business table asserts `tenant_id ∈ is_tenant_member(...)`. Tests in `tests/rls.test.ts` cover cross-tenant SELECT/INSERT/UPDATE/DELETE on representative tables.

## 14. Important Design Decisions

1. **Username + synthetic email** (`src/lib/auth.functions.ts`):
   - Splits username from email so we keep Supabase Auth's RLS primitives while presenting a "username" UX.
   - Comment in code: "The cleanest way to swap email login for username login without a multi-week edge-function rewrite is to treat `<username>@pelecanon.local` as the canonical email shape — the user never sees it."
2. **No `Database` generic on Supabase clients**:
   - Comment in `src/integrations/supabase/client.ts`: "The previous `<Database>` constraint was causing every `.from('tablename')` chain to collapse to `never` because the placeholder Database type (which can't be regenerated without the Supabase CLI) doesn't fully match the shape PostgREST's generic resolution requires."
   - Solution: omit the generic on `createClient`, keep `types.ts` as a PostgREST-compatible stub for future migration to real generated types.
3. **Service-role bypass for catalog tables**:
   - Comment in `src/lib/catalog.functions.ts`: "Several catalog tables (product_templates, materials, suppliers, ...) have Phase 1 RLS policies that the admin role can't satisfy through PostgREST — reads return 0 rows. Until those policies are loosened server-side, these fns route reads & writes through supabaseAdmin which bypasses RLS entirely."
4. **OPTIONS preflight over HEAD** (`src/components/photo-uploader.tsx:55`):
   - Comment: "Sprint 0: real OPTIONS preflight, not HEAD. HEAD on a signed-PUT URL is a CORS *simple* request — browsers don't preflight it and don't send Content-Type/Authorization, so it succeeds even when the PUT that follows would be blocked."
5. **tenant_id prefix on R2 keys** (`src/lib/r2.server.ts`, `src/lib/r2.functions.ts`):
   - Comment: "tenant scoping is the primary isolation boundary; never bypass it."
6. **Single-source pricing**:
   - `src/lib/pricing.ts` is named `// Centralized pricing engine — single source of truth. All formula logic lives here, NOT scattered across components.` Documented in README too.
7. **`Dynamic import of @tanstack/react-start/server-entry`** (`src/server.ts:14`):
   - Enables a graceful SSR error path before the fully-typed runtime is available.
8. **Append-only audit/notifications tables**:
   - `.lovable/STATUS.md` and the migrations restrict `notification_log` + `audit_log` against UPDATE/DELETE.
9. **PLC ID lifecycle** (`src/lib/numbering.ts`, `src/lib/order.functions.ts`, `src/lib/invoice.functions.ts`):
   - Quote generates a PLC ID, passes it into both `createInvoiceFromQuote` and `createOrder` so the unified number threads through the chain.
10. **Server functions over separate API routes**: `createServerFn` ensures SSR-aware RPCs without a parallel REST layer; consuming via `useServerFn` + TanStack Query gives us type-safe client SDK + cache + retries for free.

## 15. Technical Debt

1. **`.lovable/STATUS.md` flags**:
   - **`.lovable/neon-migration-plan.md` decision lock contradicts current state**: Neon cutover is documented as in progress, but `src/db/client.server.ts` still uses `postgres-js` (raw TCP) against Supabase Postgres, not `drizzle-orm/neon-http`. Comment in the file: "Renamed from the previous Neon HTTP adapter — Neon is no longer part of the stack." — but the plan file and the R2 setup expect Neon as the future path.
   - **`companies` table artefact** referenced as needing a follow-up migration (might already be done, hard to tell from current state).
   - **`configurations` table might have a `company_id` column** missed during the cascade drop (per `.lovable/STATUS.md`).
2. **`attachments` is the only table typed in Drizzle schema** (`src/db/schema.ts`) — the rest of the schema is **not yet** mirrored there, contrary to the documentation claim that "Drizzle schema in src/db/schema.ts is the source of truth." This is flagged as Phase 4 work.
3. **`useAuth` quirks** (`src/lib/useAuth.tsx`):
   - Tracks memberships via direct `supabase.from('tenant_members').select(...)` AND also via bootstrap. The initial fetch runs before bootstrap in many cases, so the first paint may show no memberships briefly.
   - `bootstrapTokenRef` race prevention relies on a numeric counter — solid, but worth a comment block for future readers.
4. **`admin.ts` unused** in client code paths (comment: `// Generic intentionally omitted for the same reason as client.ts`). File is dead weight today.
5. **`r2.config.ts` ships no runtime handling** — only type + CORS policy text. Fine as documentation, but worth a doc comment pointing to `docs/R2.md`.
6. **Retry / DLQ for n8n**: No structured handling for failures; failed notifications accumulate in `notification_log` and never re-fire.
7. **`createOrder`/`createInvoiceFromQuote` lack cleanup-on-failure**: if the second insert fails after the first succeeded, half a chain is left behind.
8. **Localization coverage holes**: English (`src/i18n/locales/en.json`) is the comprehensive one; Arabic has the admin nav + dashboard translation partial; French is the smallest subset. The fallback handler ensures the UI doesn't blank, but untranslated keys fall back to dot-separated fragments.
9. **Lack of CSRF for non-serverFn reads** (`src/start.ts:14`): CSRF middleware is filtered to `handlerType === 'serverFn'` only. Read endpoints aren't serverFn-wrapped; they go through the supabase client. Acceptable for the current SPA-only auth model, but a risk in case any admin route starts hitting `supabase.from` directly for a write.
10. **`tests/rls.test.ts` not wired to npm scripts**: `package.json` has no test script. Running requires explicit `bunx vitest run tests/rls.test.ts` and a Postgres instance reachable from `DATABASE_URL_TEST`.

12. **`tsconfig.json` strictness assumed** but not visible in context — ESLint config also excluded; conventions are inferred from code style only. If TypeScript strict mode flips later, dozens of "any" coercions in components (`any[]`, `any`) would break.

## Appendix B — Conventions inferred from code style

- **Components**: PascalCase (e.g. `AvatarUploader.tsx`).
- **Hooks**: `useCamelCase` (e.g. `useIsMobile`, `useAuth`).
- **Server functions**: kebab-case file (`pricing-factors.functions.ts`, `apply-migration.functions.ts`), but *also* allowed camelCase (`auth.functions.ts`, `seed.functions.ts`). Mixed — adopt `<entity>.functions.ts` for new ones.
- **Database columns**: snake_case in SQL; camelCase in TS (`name_ar` vs `nameAr` only appears in typed Drizzle mirrors like `tenants` table).
- **No raw `fetch` in components** (where possible) — server-fns + TanStack Query is the rule. Photo uploader is the lone exception (direct PUT to R2 by design).
- **No `any`** — Drizzle/Supabase inferred types used. InternalNotes and orders.tsx use `any[]` to bypass gaps (technical debt).
- **No `console.log` in production** — except in code that explains itself (seed.functions, materials.functions, suspiciously chatty).
- **No drafts/placeholders in code**: every code path runs end-to-end. UI strings lean on `i18next`.
- **`cn()`** from `@/lib/utils.ts` for class merging (standard shadcn practice).
- **Auth gate pattern**: server fn optional `.middleware([requireSupabaseAuth])`; UI pages check `useAuth().isStaff` before rendering.
