# Cloudflare R2 — Production Setup

This document covers everything you need to make R2 work for the
**production-photos** (public, customer-facing) and **attachments**
(private + presigned on demand) flows.

> If you change the visibility mode for any entity, the existing
> `attachments.is_public` column lets you flip individual rows without a
> migration.

## 1. Environment variables

Add the following to your `.env` for **production** AND local dev:

```
R2_ACCOUNT_ID=32hex_account_id_here
R2_ACCESS_KEY_ID=your_r2_access_key
R2_SECRET_ACCESS_KEY=your_r2_secret_key
R2_BUCKET=pelecanon                   # bucket name (lowercase, hyphens only)
R2_PUBLIC_URL=https://files.your-domain.example  # public CDN URL
```

How to get each value:
- `R2_ACCOUNT_ID`: Cloudflare dashboard → R2 → right-hand "Account ID".
- `R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY`: R2 → Manage R2 API Tokens
  → Create token. Scope: Object Read & Write; bucket: the bucket above.
- `R2_PUBLIC_URL`: if you attach a custom domain to the bucket
  (recommended), use `https://files.your-domain.example`. If the bucket
  is left public without a custom domain, use
  `https://<account>.r2.cloudflarestorage.com/<bucket>` (default).

> Tip: If the bucket is fully private, omit `R2_PUBLIC_URL`. The server
> then signs every URL (25-minute TTL) for the public tracking page.

## 2. Bucket public access (default for production photos)

For production photos going to customers on the tracking page, you have
three options:

| Mode                          | `R2_PUBLIC_URL`  | Render method                | Use when                                |
|------------------------------|------------------|------------------------------|----------------------------------------|
| Public bucket + custom domain | ✅ set           | Direct CDN URL               | Recommended for customer-facing photos |
| Public bucket (R2 default)    | ✅ set           | R2 S3 URL                    | OK for staging only                     |
| Private                       | ❌ unset         | Presigned GET (server fn)    | Recommended for attachments             |

To make a bucket public for customer-facing photos:
1. R2 → bucket → Settings → Public Access → Allow Access.

For attachments, leave them private; the `attachments` table records
`is_public=false` and render paths sign a URL on demand.

## 3. CORS policy (REQUIRED for direct browser uploads)

Browser-side PUT uploads from the admin app need the bucket's CORS
policy to allow the source domain. Paste this into R2 → bucket →
Settings → CORS Policy:

```json
[
  {
    "AllowedOrigins": [
      "https://your-production-domain.example",
      "http://localhost:5173"
    ],
    "AllowedMethods": ["PUT", "GET", "HEAD"],
    "AllowedHeaders": ["Content-Type", "Authorization"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3600
  }
]
```

If you don't apply this, browser PUTs to R2 will fail with a CORS error
even though the request reached the bucket. The signer runs a real
OPTIONS preflight probe; if R2 rejects it, you'll see:

> CORS preflight failed (HTTP 403). Confirm the R2 bucket policy allows
> PUT/GET/HEAD from `<your origin>` with the Content-Type header.

## 4. Object key strategy

The server functions in `src/lib/r2.functions.ts` write keys as
`tenant/<entityType>/<entityId>/<hash>.<ext>`. Tenant scoping is the
primary isolation boundary; **never bypass it**.

Examples:
- `pelecanon/production-photos/8f9c-…/a1b2.webp`
- `pelecanon/attachments/74e0-…/contract.pdf`
- `pelecanon/avatars/<userId>/<hash>.webp`

If you migrate to a multi-bucket strategy later, only the key prefix
stays — the rest of the codebase never sees bucket internals.

## 5. Mapping for production inferring visibility

| Entity                                  | `is_public`     | Reason                                          |
|----------------------------------------|------------------|-------------------------------------------------|
| `production-photos` rows               | implicit public  | Shown on the customer tracking page (`/track`). |
| `attachments` rows                      | default false    | Internal — admin only.                          |
| `avatars` rows                          | implicit public  | Pulled from URL by other users (team page).     |

To override per-row, flip `is_public` on `attachments` after insert.

## 6. Migration from Supabase Storage

If you're moving old `production-photos` from the Supabase Storage bucket
into R2, the canonical plan is:

1. Run `src/lib/r2-migration.functions.ts → migratePhotosToR2` once
   (dry-run first). Rows' `photo_url` becomes an R2 public URL.
2. Bucket stays public for the same reason as before.

The `extractR2Key()` helper in `src/lib/r2.utils.ts` recognizes both:
- `https://<account>.r2.cloudflarestorage.com/<bucket>/<key>`
- `https://<custom-domain>/<key>`

so the migration is idempotent — re-running it on already-R2 URLs does
nothing.
